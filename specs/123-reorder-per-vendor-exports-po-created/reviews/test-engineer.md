**Blocking finding ‚Äî `npm run typecheck:test` is red on this diff.** This is one of the two typecheck gates explicitly named in this project's testing-reality contract (CLAUDE.md: "Typecheck gates: `npm run typecheck` and `npm run typecheck:test`") and is wired as a required CI job ("Track 1a ‚Äî typecheck (test graph)" in `.github/workflows/test.yml`). The spec's own "Verification" section only lists `npx tsc --noEmit` (the *base* config ‚Äî Track 1b), which is clean; it does not mention running `typecheck:test`, and that command currently fails with 8 errors across 8 pre-existing test files that build `ReorderVendor` fixtures without a `hasPo` field. `npx jest` itself still passes (Jest via ts-jest/babel doesn't hard-fail on these particular type errors in this config), which is why the developer's full-jest run looked clean ‚Äî but the dedicated typecheck job would fail on push to `main`, per CLAUDE.md's own asymmetry warning ("a green `test.yml`... [is] NOT sufficient" cuts both ways: a green `jest` job with a red sibling `typecheck` job in the same workflow file is exactly the kind of partial-green state that needs to be caught before merge). This is a genuine build-breaking regression, not a style nit ‚Äî recommend either (a) making `ReorderVendor.hasPo` optional (`hasPo?: boolean`, matching the `db.ts` mapper's "absent ‚Üí false" default-tolerant design already documented in the type's own JSDoc: *"`false` for legacy null-reference_date drafts (never matched)"* ‚Äî the field is already treated as optional-with-default everywhere it's read via `vendorHasPo = (v) => v.hasPo ?? false`), or (b) backfilling `hasPo: false` into the 8 affected fixture files. Either fix is small; I did not make it myself per the "surface, don't fix" rule for a developer-owned regression.

Not flagged as a spec-123 pgTAP/jest gap for a Test-Engineer BLOCK because it isn't one of the two highest-risk ACs called out in the review brief, and the underlying wiring is a 3-line, easily-inspectable diff in `useStore.ts` ‚Äî but a follow-up unit test on `createPoDraft`'s call to `db.createPurchaseOrderDraft` (asserting the `referenceDate` argument) would close this cleanly.
