## Recommended next steps (ordered)
1. **[Critical] Add `unsubscribeFromPush()` teardown to the three staff sign-out call sites** (`EODCount.tsx:536`, `Reorder.tsx:490`, `Receiving.tsx:328`) so staff logout tears down this device's subscription exactly as admin logout does (`useStore.ts:753`). Fix this first — it is the only ship-blocker and it is what makes the shared-device leak reachable. **Scope decision: pull this INTO spec 118, do not split it.** Rationale: spec 118 created the exposure (it wires the first staff-side `requestPermissionAndSubscribe`), and the spec's own "Q (does logout affect this?)" resolution asserts logout teardown is already handled — that assertion is correct for admin but wrong for staff. Shipping 118 as-is would knowingly land a spec whose own design note is false on the surface it just activated. The fix is three small call-site edits mirroring an existing pattern; folding it in keeps the spec's per-device-isolation guarantee actually true rather than deferring a known data-integrity break. This does require an explicit Files-changed re-scope (the three staff screens are outside 118's current declared set) — that is a re-scope note for the PM/architect, not a silent patch, but the correct home is 118, not a sibling. (A last-resort alternative, only if the PM refuses the re-scope, is an immediate blocking sibling spec that ships before or with 118; but do not let 118 merge ahead of it.)
2. **[Should-fix] Clear the stale `message` state on re-probe** — clear inside `refresh()` once the view resolves, or derive body purely from `view`, so a transient failure message doesn't persist across mount/`visibilitychange` re-probes after the underlying condition changed.
3. **[Should-fix] Fold "no `userId` yet" into the `interactive`/disabled condition** (or surface a message) so tapping the toggle mid auth-restore gives feedback instead of a silent no-op.
4. **[Should-fix] Resolve the dead `chrome.notifications.retry` key** — either wire the retry affordance AC (c) calls for on `denied`/`error`, or drop the key from all 6 catalogs. Wiring it is preferable since AC (c) explicitly asks for "retry offered."
5. **[Should-fix] Extract a shared `useNotificationToggle(userId, translate)` hook** so the byte-for-byte-identical state/effect/handler logic lives in one place and the two message ternaries can't silently drift. (Do this after 2-4 so the shared hook captures the corrected logic once.)
6. **[Recommended, non-blocking] Add the two `StatusPill`-style render tests** (mock `probeNotificationState`/`requestPermissionAndSubscribe`/`unsubscribeFromPush`, assert pill text + press behavior for `on`/`off`/`needs-install`/`denied`) — or do the manual in-browser smoke the developer flagged — to cover the admin toggle's currently-zero render exposure and AC (c)'s per-code rendered copy. Also visually check the 3-child `switcherRow` wrap on a narrow phone (code-reviewer nit).
