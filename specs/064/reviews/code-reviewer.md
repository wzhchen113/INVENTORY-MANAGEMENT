## Code review for spec 064

### Critical

- `.github/workflows/db-migrations-applied.yml:15-16` — `on: push:` has no branch filter, so the gate triggers on every push to every branch, not just `main`. The spec says "Runs on every pull request and every push to `main`." Running on every feature-branch push consumes a management-API call to prod per push, adds unnecessary CI noise (and potential rate pressure) for branches that aren't ready for prod alignment, and is inconsistent with the stated acceptance criterion. Fix: add `branches: [main]` under `push:`, matching the intent stated in the spec. (Note: `test.yml` also omits a branch filter; that pre-existing inconsistency is out of scope here, but this workflow should not replicate it given the AC explicitly says "push to `main`.")

### Should-fix

- `.github/workflows/db-migrations-applied.yml:105-108` — The awk parser computes `/tmp/cli_local.txt` (the CLI-derived local set) but this file is never referenced after the awk block. The architect's defense-in-depth rationale (spec §Awk parser: "if the CLI's `migration list` ever drops a local file, the filesystem-derived list catches it") is correctly implemented — the `comm` comparisons use `/tmp/local_versions.txt` (filesystem-derived). However, the cross-check the architect described would be: compare the two local sources against each other and warn if they diverge. As written, `/tmp/cli_local.txt` is computed and silently discarded. This means a scenario where the CLI underreports local files is silently undetected. Suggested fix: after building both local sources, add a `comm -13 /tmp/local_versions.txt <(sort -u /tmp/cli_local.txt) > /tmp/cli_only_local.txt` cross-check and emit a warning step-summary line if it is non-empty ("CLI reported local migrations not found on disk — possible CLI version issue").

- `.github/workflows/db-migrations-applied.yml:111-112` — `sort -u /tmp/cli_remote.txt > /tmp/remote_versions.txt 2>/dev/null || touch /tmp/remote_versions.txt` silently swallows a `sort` error (e.g. read permission problem on `/tmp/cli_remote.txt`) via `2>/dev/null`. This is fine for the "file doesn't exist" case (no remote entries), but means a genuine I/O error would produce an empty `remote_versions.txt` and silently report "in sync" for direction 2. Since `set -euo pipefail` is active in the compare step, the `|| touch` should use a more targeted guard: only swallow the "file not found" error. Suggested fix: `[ -f /tmp/cli_remote.txt ] && sort -u /tmp/cli_remote.txt > /tmp/remote_versions.txt || touch /tmp/remote_versions.txt`, which preserves the error-on-I/O-failure behavior while still handling the absent-file case.

- `.github/workflows/db-migrations-applied.yml:65-70` — The `Link to prod` step has no guard for a missing `SUPABASE_PROJECT_ID` secret. If `SUPABASE_ACCESS_TOKEN` is present (non-fork) but `SUPABASE_PROJECT_ID` is unset or empty, the supabase CLI will receive `--project-ref ""` and produce an unhelpful error message. The pre-merge action items in the spec note this, but a `[ -z "$SUPABASE_PROJECT_ID" ]` check inside the step — mirroring the `SUPABASE_ACCESS_TOKEN` guard — would give an engineer a clearer error message during the repo-setup phase. This is a UX issue, not a correctness issue. Suggested fix: add a parallel `SUPABASE_PROJECT_ID` check in the secret-availability step (or a separate inline guard at the top of the link step).

### Nits

- `.github/workflows/db-migrations-applied.yml:193-204` — The "both directions clean" success block is guarded by a redundant double-negative check (`if [ ! -s /tmp/local_not_in_remote.txt ] && [ ! -s /tmp/remote_not_in_local.txt ]`). By the time execution reaches this point, `$HARD_FAIL -eq 1` would have already exited and direction-2 is irrelevant to the exit path. The check is logically correct but structurally redundant — an unconditional `else`-equivalent would be cleaner. Minor clarity nit; not a correctness issue.

- `.github/workflows/db-migrations-applied.yml:148` — Recovery command in the step summary says `npx supabase db push` but the spec's §Failure UX example says `npx supabase db push --linked`. The `--linked` flag ensures the CLI targets the linked prod project rather than requiring the developer to have previously run `supabase link` in their local checkout. Omitting it is functional (the CLI uses the linked project if the `.temp/project-ref` file exists), but the recovery command will silently fail for engineers who haven't linked locally. The spec explicitly includes `--linked` in the example command.

- `CLAUDE.md:91` — The updated "Gaps and unknowns" CI bullet no longer mentions the `FORCE_JAVASCRIPT_ACTIONS_TO_NODE24` opt-in that was called out in the original spec 047 bullet. The original text ("Spec 047 added the `FORCE_JAVASCRIPT_ACTIONS_TO_NODE24` workflow-level opt-in") was historically useful for future maintainers understanding why that env var exists. The replacement bullet drops it. Not wrong, but the spec 047 attribution for the opt-in is now unanchored in documentation. (out-of-scope) — worth considering in a future CLAUDE.md polish pass.
