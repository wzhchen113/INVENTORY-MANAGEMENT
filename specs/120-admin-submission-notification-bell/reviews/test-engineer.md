  - Caveat: the emitter's **exception-safety** (a notification failure must not roll back the submission — spec §3/§10, one of the two named highest-risk surfaces per the dispatch prompt) is **not exercised by the shipped 11-arm file at all** — no arm forces an internal failure inside `emit_submission_notification`. Recommend backend-developer add one arm (e.g., call `emit_submission_notification` directly with a bogus actor/store and assert `lives_ok` + zero rows land) — cheap to add, closes a real gap on a named risky surface.
- AC6: Per-viewer read state (independent per admin/super_admin, idempotent re-read) → **NOT TESTED** — the shipped pgTAP file has **zero arms** touching `public.notification_reads` (no RLS test, no independence test, no idempotency test), despite spec §9 explicitly calling this out as an intended arm ("Per-viewer read idempotency: two admins on the same notification track read independently; a second `markNotificationRead` is a no-op"). - **Biggest real gap: frontend jest coverage is zero for this entire feature.** `NotificationBell.tsx`, `useSubmissionNotifications.ts`, and the `useStore.ts` `submissionNotifications`/`submissionUnreadCount` slice (load, mark-one-read optimistic+revert, mark-all-read optimistic+revert) have no test file at all. I manually confirmed the DB-side behavior (RLS, independence, idempotency, the two RPCs) is correct via an ad-hoc probe, but there is no committed regression test for it.
- **pgTAP gap: emitter exception-safety (§3/§10) has zero shipped coverage.** This is one of the two surfaces the dispatching prompt called out by name as highest-risk. I confirmed correctness by hand; there is no committed regression test.

### Verdict guidance for release-coordinator

Per the dispatch instruction to block only if a HIGH-RISK AC is genuinely uncovered across all tracks (naming RLS scoping + emitter-safety as the risky surfaces, with pgTAP as their gate): **RLS brand-scoping (AC4/AC10) is solidly covered and green (arms 1–4).** Emitter exception-safety and per-viewer read-state (AC6/AC7) are **not covered by the shipped suite**, but I hand-verified both are functionally correct via ad-hoc probes against the live local DB before writing this report — so I am not recommending a BLOCK on those grounds alone. I am, however, flagging all of the above as real, currently-uncovered gaps (zero jest for the entire frontend surface; zero pgTAP for read-state and exception-safety; zero smoke for the edge function) that should be closed before this is considered fully regression-safe, and I'd treat "NOT TESTED" ACs 1, 2, 3, 6, 7, 8, 9 as open items for the release-coordinator to weigh explicitly rather than silently pass through.
