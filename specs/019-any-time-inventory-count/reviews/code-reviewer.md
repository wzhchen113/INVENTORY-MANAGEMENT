## Code review for spec 019 (Any-time inventory count)

### Critical

- `src/screens/cmd/sections/InventoryCountSection.tsx:160,274` — **Submit only collects entries from the active category filter slice, silently dropping all other filled-in rows.** `nonBlankCount`, `hasNegative`, and the `entries` array in `onSubmit` all derive from `filteredItems`, which is `storeInventory` filtered to `selectedCategory`. A user who fills in "Dairy" items, then switches the chip to "Proteins" and clicks SUBMIT will lose every dairy entry with no warning — the RPC will succeed with only protein items, and the success toast will quote the wrong item count. The footer at line 770 also shows `{nonBlankCount}/{totalItems}` scoped to the filter slice, creating a false impression of completeness. Fix: `onSubmit` must build `entries` from `storeInventory` (all items), not `filteredItems`. `nonBlankCount` / `totalItems` in the footer and tab-strip right slot should likewise be computed against `storeInventory`. `hasNegative` should guard across all items too, so a negative in a hidden category can't slip through.

### Should-fix

- `src/hooks/useRealtimeSync.ts:40` — **Dead subscription for `inventory_counts` fires `onSync` (global reload) on every insert, but no store slice consumes inventory counts.** The developer added `inventory_counts` to the `store-{storeId}` channel here AND the section opens its own `inv-count-section-{storeId}` channel (line 184 of the section file). The two are on separate named channels so Supabase will deliver two WAL events per insert. The `useRealtimeSync` one triggers a full `loadFromSupabase` call that does no useful work (no counts slice to reload). Architect §7 explicitly chose Option A (section owns the subscription) and noted the global reload is a no-op for counts — the `useRealtimeSync` addition is therefore redundant. Remove line 40 from `useRealtimeSync.ts`, keeping only the section-level subscription.

- `supabase/migrations/20260513000000_inventory_counts.sql:347` — **`coalesce(v_entry.notes, '')` stores empty string instead of NULL for no-note entries.** The column is `text null` (line 112) and the schema convention across `notes` fields in this project is to store NULL (not `''`) for the absent case — see `inventory_counts.notes` itself (no coalesce on insert, line 298-299). Storing `''` vs `NULL` inconsistently makes downstream `WHERE notes IS NOT NULL` queries behave differently for RPC-inserted rows vs raw inserts. Fix: change to `v_entry.notes` (pass through as-is; NULLs arrive as NULL from `jsonb_to_recordset`).

- `src/screens/cmd/sections/InventoryCountSection.tsx:184` — **Section-level realtime channel uses the non-standard naming `inv-count-section-{storeId}` instead of the documented `store-{id}` / `brand-{id}` convention.** CLAUDE.md lists the naming convention; `useRealtimeSync.ts` comments document the two channels as `store-{storeId}` and `brand-{brandId}`. A per-section channel is a valid Option A pattern, but the name should follow a readable convention. Given that line 40 in `useRealtimeSync.ts` should be removed (see above), the section's channel is the only subscriber and its name can be a simple variant like `store-${storeId}-inv-counts` to stay consistent with the store-scoped naming scheme.

- `src/store/useStore.ts:1428-1435` — **`client_uuid` is minted on every call, so a network-fail retry from the UI generates a new UUID and loses idempotency protection.** The store action mints a fresh UUID on each invocation. The section's submit button calls the action exactly once per press, sets `submitting = true`, and waits. If the network drops mid-flight and the RPC actually persisted the row, the user who clicks Submit again (or triggers any retry) gets a second UUID and a second row. The architect's §6 and §10 design note says "Minted ONCE per submit click; re-clicking with the same UUID returns `{ conflict: true }`" — but the current implementation mints a new UUID on each call, so re-clicking always gets a new UUID, not the same one. Fix: the section component should mint `client_uuid` before calling the store action (or persist it across retries) so a retry re-uses the same UUID. The store action should accept it as a parameter (matching the `db.ts` helper signature which already takes `clientUuid`).

### Nits

- `src/screens/cmd/sections/InventoryCountSection.tsx:315-319` — `countedAtLocal` is not reset after a successful submit. After submitting, the datetime input still shows the original timestamp. The next count will carry the old time as a pre-fill until the user manually edits it. Adding `setCountedAtLocal(localNowForInput())` to the clear block would reset it to "now" consistently with EOD's "clear = fresh start" UX.

- `src/lib/db.ts:679` — `countId: result.count_id || ''` produces a silent empty string when the RPC returns an unexpected shape. An empty `countId` in the return value would make the caller's `if (result.conflict)` path display the wrong toast (it checks `conflict` flag, fine) but would silently succeed with an unusable count ID. A `throw` or at minimum a `console.warn` on `!result.count_id` would be safer.

- `src/types/index.ts:276` — `InventoryCount.entries` is typed as non-optional `InventoryCountEntry[]`. `fetchRecentInventoryCounts` returns `InventoryCountSummary[]` (no `entries`), and `fetchInventoryCount` returns the full `InventoryCount`. The asymmetry is fine, but `InventoryCount` could be split into a base type and the detail type (or use `entries?: InventoryCountEntry[]`) to be precise — the field is documented "populated by fetchInventoryCount only" (comment on the same line), acknowledging callers that get an `InventoryCount` without entries would see an empty array, not `undefined`. Low-impact since the only consumer is the detail view, which only uses `fetchInventoryCount`. Out-of-scope suggestion, purely a type-expressiveness nit.

- `src/lib/db.ts:691` — Spec AC §`db.ts` names the function `listInventoryCounts`; architect §5 revised it to `fetchRecentInventoryCounts`. The implementation follows the architect's name. Minor inconsistency between the acceptance criteria checkbox and the shipping name — the AC checkbox at spec line 116 will remain unmatched. Not a functional issue but worth noting so the release coordinator knows the AC name-check is intentional divergence, not a gap.

- `src/screens/cmd/sections/InventoryCountSection.tsx:9` — `import { supabase } from '../../../lib/supabase'` is used only for `supabase.channel()` / `supabase.removeChannel()` (realtime subscription management), not for `.from()` or `.rpc()`. CLAUDE.md's "Direct Supabase calls" prohibition targets PostgREST/RPC, so this is technically compliant. However if the `useRealtimeSync` redundant subscription is removed (see Should-fix above), consider whether the section's subscription could be moved into a small custom hook (similar to `useRealtimeSync`) to keep the raw `supabase` import out of the section and in `src/hooks/`. Low priority.
