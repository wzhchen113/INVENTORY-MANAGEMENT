## Known tension — the `product_page_url` editor field (owner is the tiebreaker)
Two reviewers reached opposite conclusions on the editor field that was built despite the design deferring it. Both are legitimate; this is a scope-authority call the owner must make.

- **backend-architect: KEEP.** The field is harmless + additive, threaded identically to spec-114's reviewed `order_code` pattern; the column + read-mapper stay either way (D-7 required them), so reverting only the editor hunks leaves the column populated exclusively by direct-DB writes — a strictly worse operator experience. It gives spec 132 a head-start, not a conflict. Condition: spec 132 must NOT re-build the field, and the `section.inventory.productPageUrl*` i18n keys now live in the 131 catalog delta.
- **code-reviewer: REVERT (pending explicit re-approval).** The design (D-2, D-13) and the spec file — the pipeline's source of truth — twice say "NOT built here." A developer-side resolution against an explicit "NOT built here" should not stand without architect/owner re-approval. Revert the `IngredientForm.tsx` / `IngredientFormDrawer.tsx` editor hunks + the `productPageUrl*` i18n keys; keep only the column + `db.ts` mapper/upsert threading (unambiguously D-7 in scope). Land the editor field with spec 132 as originally planned.

**Recommendation:** This is not a correctness defect either way (security clean, no AC depends on it — 131's ACs only cover `extension_ordering`/`order_page_url` on `vendors`). Surface it to the owner as a binary decision. Because the two reviewers split and the architect's KEEP is conditioned on spec-132 coordination, the safest default absent an owner call is code-reviewer's REVERT — it restores design/spec alignment with the smallest surface and defers cleanly. **But if the owner prefers KEEP** (a legitimate call), it must be paired with the `apply_item_vendors_to_brand` propagation fix (code-reviewer finding 2) in the same PR, otherwise admins get a silent order-code-propagates / product-URL-doesn't inconsistency.
