### Notes

- **AC-6 gap (the main finding):** the mark-ordered write-back's store-scoping is untested. Concretely: nothing in the pgTAP suite proves that a non-member's `update purchase_orders set status='sent' where id=:poId and status='draft'` (the exact mechanism D-4 specifies, and the one both the extension and the pre-existing in-app "mark as sent manually" button use) is denied by `store_member_update_purchase_orders`. This matters more than a typical missed case because CLAUDE.md and this project's own testing culture treat "RLS denial tests are as important as RLS allow tests" as a hard rule, and the read-side of this exact feature (RPC 1 `(P5)`, RPC 2 `(Q2)`) got exactly this kind of denial test — only the write-side was skipped. Recommend the backend-developer add one `throws_ok`/0-row-no-op case to `extension_ordering.test.sql` (bump `plan(18)` → `plan(19)`) using the existing Charles-store PO fixture already created for `(P5)`/`(Q2)`: attempt the guarded UPDATE as the Frederick (non-member) role against `test.po_charles`, assert 0 rows affected / status unchanged.
- **Secondary observation (not a spec-131 regression, but relevant to AC-6's "idempotent, store-scoped" framing):** the actual in-app "admin marks sent" code path, `db.ts::markPurchaseOrderSent` (pre-existing, spec 107, untouched by this build), issues an UNGUARDED `update purchase_orders set status='sent' where id=:poId` — no `and status='draft'` clause. The design (D-4) frames the guarded UPDATE (`...and status='draft'`) as the mechanism that makes the write idempotent and non-resurrecting for `received`/`cancelled`/`partial` POs; the pgTAP tests that exact guarded SQL shape directly (not through `db.ts`), so the *pattern* is proven correct, but the concrete function the in-app admin button calls does not itself carry that guard. In practice this is masked by `POsSection.tsx`'s client-side `canSend = selStatus === 'draft'` gating (the button doesn't render for non-draft POs), so it is unlikely to be user-triggerable today, but it means the DB-level idempotency/non-resurrection guarantee AC-6 describes is not actually enforced for that one caller — only client-side. Flagging for awareness; not blocking this spec since the code is pre-existing and out of this build's slice ownership (D-13 didn't scope touching `db.ts::markPurchaseOrderSent`).
- **Scope drift (already self-flagged by the developer, confirmed on inspection):** the `item_vendors.product_page_url` *editor* field in `IngredientForm`/`IngredientFormDrawer` was built even though design D-2/D-13 explicitly deferred it to spec 132/follow-up. It is not part of any spec-131 AC (spec 131's ACs only mention `extension_ordering`/`order_page_url` on `vendors`), so it does not affect AC pass/fail here, but it has no dedicated behavioral test — `IngredientForm.test.ts`'s diff is exclusively type-forced fixture-literal updates (`productPageUrl: ''` added to existing `VendorLinkRow` literals), not a new test of the field's own change/trim/null-on-empty handling. Architect reconciliation was explicitly requested by the developer; test coverage should follow whatever that reconciliation decides (keep-and-test, or revert).
- **Migration filename deviation** (`20260723000000_extension_ordering.sql` instead of the design's asserted `20260720000000`) is cosmetic and self-flagged; column/RPC contents are unchanged from the design and verified correct by direct read + passing pgTAP.
- **Shell-smoke track:** the design marked this optional (D-12) and none was added. No gap — optional tracks with no test are not a BLOCK per spec 022's own posture.
- **Realtime / publication-restart step:** correctly not required and not performed — no `supabase_realtime` publication membership changed by this migration (verified by reading the migration; only column adds to already-published tables).
