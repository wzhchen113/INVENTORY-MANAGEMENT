### Critical

### Should-fix

- `src/screens/cmd/sections/EODCountSection.tsx:613-631` — Comment says the `+ vendor` button is "Disabled when we're showing unscheduled vendors (the schedule is the wrong surface to mutate from there) or in __all__ (defensive)." Neither condition is implemented: the `TouchableOpacity` has no `disabled` prop, no `opacity` change, and no `pointerEvents` guard for either case. In the `showUnscheduled` ON state the modal still fires, which lets a user add vendors to the schedule while "showing all" — the intended restriction from the comment goes unenforced. Fix: add `disabled={showUnscheduled}` and `opacity: showUnscheduled ? 0.4 : 1`, consistent with the REST-day button treatment at lines 492-503.

- `src/screens/cmd/sections/OrderScheduleSection.tsx:179` — Inline color literal `'#000'` for the checkmark text on a filled accent cell. In dark mode the accent background is `'#7DD668'` (light green) so black text is readable today, but the value isn't routed through `useCmdColors()`. If the accent token ever changes, the checkmark could become invisible. Fix: use `C.bg` (the background color token) or a dedicated `C.onAccent` token — `C.bg` is already `'#08090C'` in dark mode and `'#FAFAF8'` in light, so swap to the inverse of the cell background: `color: C.bg` for the ✓ glyph when `isOn`.

- `src/lib/db.ts:1532-1566` — The comment at line 1532-1537 says `addOrderScheduleEntry` treats a 23505 unique-violation as an idempotent no-op "so double-clicks and stale optimistic state don't surface as backend errors." But the comment on the migration also notes the constraint was added `ON CONFLICT DO NOTHING`. Those are two different behaviors. `ON CONFLICT DO NOTHING` is not what the INSERT uses — it uses a plain `INSERT`, so the 23505 is only swallowed client-side. This is correct and intentional, but the migration comment (`ON CONFLICT DO NOTHING`) reads as if that clause is in the SQL, while the actual SQL in `addOrderScheduleEntry` does NOT use `ON CONFLICT` — it uses plain `INSERT` and swallows the PG error code post-hoc. The discrepancy between the migration comment and the actual helper behavior is not a bug (the behavior is correct) but it's misleading. Fix: update the comment in `db.ts:1532-1537` to say "plain INSERT; 23505 caught client-side" instead of referencing `ON CONFLICT DO NOTHING`.

- `src/store/useStore.ts:1057-1078` — `addOrderScheduleEntry` in the store guard-checks `vendor.vendorId` before firing the db call (`if (storeId && storeId !== '__all__' && vendor.vendorId)`), but the optimistic set at lines 1059-1071 runs unconditionally even when `vendor.vendorId` is falsy. This means a vendor with no `id` (a pathological case) would get appended to the slice with `vendorId: undefined`, making it permanently invisible to `dayScheduledVendorIds` (which filters out falsy ids). The optimistic write would stick with no backend call and no revert. Fix: guard the optimistic `set(...)` with the same `vendor.vendorId` check, or move the guard to before the optimistic mutation.

### Nits

- `src/screens/cmd/sections/EODCountSection.tsx:588`, `src/screens/cmd/sections/OrderScheduleSection.tsx:67` — Null-guards on Zustand store action selectors (`if (!removeOrderScheduleEntry) return;`, `if (!addOrderScheduleEntry) return;`). Zustand selectors always return the value (they'd throw before reaching this point if the action weren't defined), so these are unreachable. They don't cause bugs, but they add noise and could mislead a reader into thinking these are optional. Drop them.

- `src/screens/cmd/sections/EODCountSection.tsx:479` — `new Date().toLocaleDateString(...)` in the TabStrip `rightSlot` still derives the display date from `new Date()` without `localDayIso`. This is display-only (not used for any computation or comparison), so the UTC-crossing issue doesn't apply — but it is inconsistent with the rest of the file's pattern. A reader might wonder whether the fix missed it. A brief inline comment ("display-only; UTC ≠ local TZ for this format is acceptable") would close that ambiguity.

- `src/screens/cmd/sections/EODCountSection.tsx:85` — `showUnscheduled` is component-scoped state that persists across day changes. If a user enables "show all" on Thursday and then clicks Wednesday, the toggle stays on and the "no vendors scheduled for wednesday" empty state never renders. The spec doesn't require a reset on day change and the toggle-is-session-global design is intentional per Q4=(d), but a brief comment noting this is intentional would prevent future "fix" attempts.

- `src/screens/cmd/InventoryDesktopLayout.tsx:129` — (out-of-scope) `todayStr = new Date().toISOString().slice(0, 10)` is the same UTC-based pattern that `EODCountSection`'s fix-pass just corrected. This instance drives the footer EOD count badge (`submittedToday/{stores.length}`) — display-only, low stakes — but it was the same class of bug. Worth a follow-up clean-up pass to use `localDayIso`.

- `src/store/useStore.ts:1052-1056` — The JSDoc block comment describing the new actions reads slightly ambiguously: "per §3" is fine inside the spec, but `addOrderScheduleEntry` / `removeOrderScheduleEntry` is the name used in both `db.ts` and `useStore.ts`. The spec's build notes flag this as a deviation from the architect's `addScheduledVendor` / `removeScheduledVendor` names (§6). Since both the backend and frontend dev landed the same name, and it's a one-line rename if reviewers prefer the architect's name, this is low priority — but worth the release-coordinator surfacing to the user for a naming decision before the action names propagate further.

- `supabase/migrations/20260507214842_spec007_order_schedule_unique.sql:54-61` — The dedup `DELETE ... USING` inside a `DO $$ ... $$` block assigns the result into `dups` via a CTE but then `SELECT count(*) INTO deleted_count FROM dups`. In PostgreSQL, a `DELETE ... RETURNING` inside a CTE is executed regardless of whether the outer `SELECT` reads it, so the delete runs correctly. But the pattern is slightly surprising — the count reporting requires the `dups` CTE to be consumed; a reader might expect `deleted_count` is inaccurate if rows were deleted but the CTE wasn't materialised. In practice PG does materialise CTEs by default (pre-PG 12 always, PG 12+ with `MATERIALIZED` keyword needed to force it; in PG 17 the default is inline but DML CTEs are always materialised). This is fine, just worth a one-line comment: `-- DML CTEs are always materialised in PG; count is accurate`.
