### Should-fix

- `supabase/migrations/20260530000000_record_missed_orders_rpc.sql:252-254` — Cron body uses `'UTC'` instead of the architect's explicitly designed `'America/New_York'`. The design §pg_cron schedule (spec line 542) specifies `(now() at time zone 'America/New_York')::date - 1` and its rationale (lines 554-556) explains this is "yesterday in NY-local terms." The implementation uses `((now() at time zone 'UTC') - interval '1 day')::date` ("yesterday in UTC"). At 07:00 UTC these coincidentally produce the same calendar date (since 07:00 UTC = 03:00 ET, NY date = UTC date), but the semantics diverge if the cron schedule is ever shifted earlier (e.g. 05:00 UTC = 00:00-01:00 ET, where the NY date may be one day behind the UTC date). The fix is one word: replace `'UTC'` with `'America/New_York'` in line 253 to match the architect's contracted computation.

- `supabase/tests/missed_order_audit_rpc.test.sql` — Two arms from the spec's §pgTAP coverage acceptance criteria are absent from the test file:
  1. Spec arm D (spec line 88-91): "The vendorName-ilike match is case-insensitive (matches the `cmdSelectors.ts:895` predicate)." No test seeds `order_schedule.vendor_name = 'us food'` with a `vendors.name = 'US FOOD'` and verifies the NOT EXISTS suppression fires. This is a case-sensitivity correctness guarantee the architect explicitly called out in §"Risks and tradeoffs / Vendor name normalization."
  2. Spec arm E1 (spec lines 762-766): "vendor_id NULL fallback — Seed `order_schedule (S, 'Monday', vendor_id=NULL, vendor_name='ACME')`, assert the inserted `item_ref = 'vendor:ACME'` (NOT `'vendor:'` empty string)." This guards the `coalesce(os.vendor_id::text, os.vendor_name)` expression which would silently produce `'vendor:'` if `vendor_id` is NULL and `vendor_name` is also NULL or if the coalesce is accidentally dropped. These are spec-required acceptance criteria, not stretch goals.

### Nits

- `supabase/migrations/20260530000000_record_missed_orders_rpc.sql:93-94` — Migration header comment says "per spec 057's `20260514140000_realtime_publication_tighten.sql:42-53`" but spec 057 is `use-connection-status-hook`, unrelated to publication membership. The realtime tighten migration carries no spec number in its own header. Either drop the "spec 057" attribution or verify the correct spec number before merging.

- `supabase/migrations/20260530000000_record_missed_orders_rpc.sql:224` — Grant block uses `revoke execute on function` where the spec acceptance criteria (line 74) and architect's design (line 517) both specify `revoke all on function`. For PostgreSQL functions EXECUTE is the only privilege so these are semantically identical, and `revoke execute` is actually the codebase-dominant convention (22 occurrences vs 2 for `revoke all`). Discrepancy is harmless — leaving as a nit for future spec authors who copy-paste the spec's grant block verbatim.
