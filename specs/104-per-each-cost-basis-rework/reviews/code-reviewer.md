## Code review for spec 104

### Critical

- `src/screens/cmd/sections/WasteLogSection.tsx:82` — `previewCost = pickedItem ? qtyNum * pickedItem.costPerUnit : 0` is missing the `× subUnitSize` bridge. `qtyNum` is in counted units; `costPerUnit` is now per-each. For a `subUnitSize > 1` item (e.g., the 2000-pack cup) the live preview shown to the user before they submit reads `sub_unit_size×` too low. The spec's governing constraint ("every consumer-visible dollar figure stays unchanged") applies to this display too. Fix: `qtyNum * pickedItem.costPerUnit * (pickedItem.subUnitSize || 1)`. The waste row's stored dollar is correct (the `logWasteEntry` write-side bridge is applied server-side), but the on-screen preview before submit is wrong.

### Should-fix

For `subUnitSize > 1` items (e.g., a packaging item tracked in `each` with `subUnitSize 2000`), the display reads `$0.02/each` where `each` is the tracking unit, but the value IS the per-each (smallest-unit) cost. ### Nits

- `src/utils/unitConversion.ts:309-312` — `calcUnitCost` guards `caseQty <= 0` before calling `piecesPerCase`, then checks `pieces > 0` in the return. The second `pieces > 0` guard is redundant: `piecesPerCase` always returns `≥ 1` (both factors default to 1), so after the `caseQty <= 0` guard `pieces` is always positive. The extra check is not wrong but could mislead a reader into thinking `piecesPerCase` can return 0. A comment noting "pieces is always ≥ 1 here (piecesPerCase floors both factors to 1)" would clarify.

- `src/lib/db.ts:690-698` — The `logWasteEntry` waste-snapshot lookup performs a separate `supabase.from('inventory_items').select(...)` round-trip to fetch `sub_unit_size` on every waste submit. The call is inside the `track()` guard and uses `abortSignal`, which is correct form. The comment explains the intentional design choice (single indexed read on a rare write). No action needed; calling this out as a potential future optimization target if waste volume grows.

- `supabase/migrations/20260701000000_spec104_per_each_cost_basis.sql:145-157` — The `spec104_per_each_cost_audit` table uses `primary key (source_table, row_id)`. However, the table definition lists the columns in `(source_table, row_id, …)` insertion order but the PK constraint puts `source_table` first — which is fine, but worth noting the INSERT and PK key order are consistent (they are). No action needed; added for clarity of intent.

- `src/screens/cmd/sections/InventoryCatalogMode.tsx:402-404` — The `avgCost` fallback (`g.rows.reduce(…) / Math.max(1, g.rows.length)`) feeds the `perEachCost` function's `costPerUnit` fallback.
