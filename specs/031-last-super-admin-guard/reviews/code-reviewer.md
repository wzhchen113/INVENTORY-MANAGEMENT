## Code review for spec 031

### Critical

_None._

### Should-fix

- `src/lib/auth.ts:109-127` (interaction with `supabase/functions/delete-user/index.ts`) — The spec's §9 states that a server-side HTTP 400 from `assert_not_last_of_role` "will surface as a toast" via `notifyBackendError`. This is inaccurate. `callEdgeFunction` uses a bare `await fetch(...)` with no `response.ok` check and a blanket catch that swallows all errors. A 400 response from the edge function does not throw; `callEdgeFunction` resolves normally; `deleteUser` returns `{ error: null }`; `deleteProfile` returns `true`; the local user cache is updated as if the delete succeeded; and a success toast fires. The user row reappears on the next refresh. This pre-dates spec 031 (the self-delete 400 has the same silent path), but spec 031's §9 actively misstates the behavior and anyone relying on the toast-on-server-refusal path will find it missing. The client-side guard (`lastOfRole`) is the primary UX signal in practice, but the silent-success behavior on a bypassed guard is a latent UX bug. Fix: add a response-status + body-parse check in `callEdgeFunction` (or a dedicated `callEdgeFunctionChecked` variant for destructive operations) that throws on non-2xx, making `deleteUser` return an error and `deleteProfile` toast correctly.

- `supabase/tests/delete_last_privileged_guard.test.sql:38-41` — The plan-count comment says "No fixture sanity assertion — the seed UUIDs are literals embedded in the do$$ block; a fixture assertion is unnecessary here." This is a reasonable choice, but "unnecessary" understates why the pattern differs from the sibling `invitations_super_admin_rls.test.sql` (which uses `isnt(current_setting('test.admin_id', true), '', ...)` as its 4th plan arm precisely to catch a broken fixture). The new test opts for plan(4) with no sanity arm, relying on the fact that the UUIDs are literal constants rather than `select id from ... where email = ...` lookups that could return empty. The comment at line 39 should make this explicit ("Because the UUIDs are compile-time literals, not runtime lookups, there is no drift risk and no sanity arm is needed — unlike the invitations sibling.") so a future reader doesn't add a sanity arm and break plan(4) without adjusting `select plan(4)`.

### Nits

- `supabase/migrations/20260514160000_assert_not_last_of_role.sql:71` — The `else format('cannot delete the last %s', target_role)` branch is unreachable. The function returns early on line 55 for any `target_role` not in `{'super_admin', 'master'}`, and the preceding two `when` arms are exhaustive for the two remaining values that reach the `case`. A dead `else` implies a reachable path that does not exist and will confuse a future maintainer adding a third guarded role who now wonders whether the `else` fires. Either remove it (`end case;`) or replace it with an explicit `ASSERT false, 'unreachable';` if you want belt-and-suspenders safety.

- `supabase/functions/delete-user/index.ts:84-89` — The spec pseudocode (§4) included a `lookupError.code !== 'PGRST116'` carve-out. The implementation omits it (treats any `lookupError` as HTTP 500). This is correct: `.maybeSingle()` does not emit `PGRST116` for zero rows — it returns `data=null, error=null`. But the deviation from the spec pseudocode is silent. A one-line comment such as `// .maybeSingle() returns data=null, error=null for zero rows — no PGRST116 guard needed` would prevent a future reader from "restoring" the spec pseudocode and introducing a dead branch.

- `src/screens/cmd/sections/UsersSection.tsx:284-288` — The two `lastOfRole` guard clauses (`&& !(user.role === 'super_admin' && lastOfRole.super_admin)` and `&& !(user.role === 'master' && lastOfRole.master)`) are unreachable for non-master callers. When `isMaster === false`, the base predicate on line 286 already forces `canDelete = false` for any row whose `user.role` is `'master'` or `'super_admin'`. The spec-031 guard only has observable effect when `isMaster === true`. The comment at lines 280-283 mentions spec-031 without this caveat, implying the guard applies generally. A parenthetical "(effective only when isMaster; non-master callers already exclude super_admin/master rows via the base predicate)" would prevent a future spec from omitting the base predicate and accidentally widening non-master delete permissions.
