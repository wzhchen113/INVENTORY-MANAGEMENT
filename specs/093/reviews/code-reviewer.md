### Should-fix

- `src/utils/unitConversion.ts:207-210` — The `convertToItemUnit` JSDoc "Resolution order" description at line 208 still says `"use item.caseQty × item.subUnitSize to get total sub-units per case"`. This is the exact formula spec 093 repudiates for `calcUnitCost`; it will mislead a reader into thinking Case-3 multiplies both axes together. The actual implementation at line 244-254 correctly uses `subUnitSize` alone (and the inline comment there explains why). The JSDoc description should be updated to match the implementation — e.g. `"use item.subUnitSize to convert sub-units into the tracking unit"` — so the two levels of documentation agree. The current state leaves an incorrect specification comment one git-blame away from re-introducing the 12× error.

- `scripts/smoke-migrate-spec093.sh:33` — The script uses `set -u` but not `set -o pipefail` or `set -e`. Comparing to the existing smoke scripts: `smoke-multi-brand.sh` uses `set -euo pipefail`; `smoke-rpc.sh` and `smoke-edge.sh` intentionally omit `-e` with an explicit comment explaining why (they want to capture failed curls and report cleanly). This smoke script collects results through a `$FAILED` flag pattern (same as `smoke-rpc.sh`/`smoke-edge.sh`) and exits early on a fatal error, so the `set -u`-only posture is defensible and consistent with those peers. However, `psql_val` is used via `$(...)` command substitution in several checks — if the `docker exec` call itself fails (container vanishes mid-run), the exit code is silently swallowed by the assignment. Adding `set -o pipefail` at minimum would catch piped-subcommand failures in the `docker exec -i ... -f -` apply step. Suggest adding `set -o pipefail` to match `smoke-multi-brand.sh`'s defensive posture, or add an explicit comment (like `smoke-rpc.sh:30`) documenting why `-e` is omitted so future maintainers don't add it thinking it was accidentally left out.

- `src/components/cmd/IngredientForm.tsx:802` — The readback `contentsUnit = values.subUnitUnit || values.unit || 'each'` hardcodes the noun "case" on the left side of the equation unconditionally, even when the DEFAULT UNIT itself is already a physically meaningful unit (e.g. `lbs`). For an item tracked in `lbs` with `case_qty=20` and no `subUnitUnit`, the readback reads `"1 case = 20 lbs"` — but the tracking unit IS lbs, so the case wrapper is named by the literal string "case" rather than deriving from any field. This is the behavior the spec §9 note for FE explicitly describes as the intended fallback: `"using a literal 'case' noun"`. So the implementation is per-spec. The finding here is narrower: the `contentsUnit` variable name suggests it holds what comes after "=", but it is derived from `subUnitUnit || unit`, where `unit` is the DEFAULT UNIT (the tracking unit), not a "contents" unit. For a Togo-Box-style item (DEFAULT UNIT=`each`, no `subUnitUnit`, `caseQty=450`), the readback correctly says `"1 case = 450 each"`. But for an item where the tracking unit is `cases` itself (e.g. DEFAULT UNIT=`cases`, no `subUnitUnit`), the readback would say `"1 case = {caseQty} cases"` — grammatically odd. The spec test fixtures don't exercise this configuration. This is a minor semantic issue and the spec explicitly deferred the noun-source decision to FE, so it is not a contract violation; but a brief inline comment distinguishing the two branches of the `||` chain would help maintainers understand why `unit` (the tracking unit, not a contents unit) is used as the fallback.

- `src/utils/unitConversion.ts:290-297` — The intermediate variable `const totalPerCase = caseQty;` is now a single-value alias with no computational purpose: it is assigned `caseQty`, then tested `totalPerCase <= 0`, then used as the divisor. The pre-spec-093 version needed the alias because it stored `caseQty * subUnitSize`. Post-fix, the alias adds a line of indirection without clarity benefit. Suggest inlining to `if (caseQty <= 0) return 0; return casePrice / caseQty;` (matching the db.ts:3719 fallback which already reads `return caseQty > 0 && cp > 0 ? cp / caseQty : 0`). This is a readability nit; the behavior is correct.

### Nits

- `src/components/cmd/IngredientForm.tsx:718` — The `units / case` label help string reads `"how many tracking units come in one case (e.g. 20 lbs per case)"`. The example `"20 lbs per case"` is consistent with the spec §9 copy, but `lbs` is a unit, not a count of tracking units — so the example slightly contradicts the label's "tracking units" framing. A simpler example like `"e.g. 20 per case"` or `"e.g. 450 boxes per case"` would avoid the semantic overlap. Minor wording nit; not a behavior issue.

- `supabase/migrations/20260602120000_spec093_case_qty_backfill.sql:67-68` — The `alter table ... enable row level security` and `revoke all ...` run outside the explicit `begin;`/`commit;` block at lines 45 and 126. In Postgres, DDL is transactional, so these statements ARE inside the implicit transaction covering the whole script run (the `begin;` on line 45 is explicit, and the `revoke` on line 68 is before `commit;` on line 126). On re-run, `create table if not exists` is a no-op so the `alter` and `revoke` execute again — both are idempotent, so this is not a correctness bug, just a layout curiosity. A brief comment noting that `alter table` and `revoke` are inside the transaction (despite appearing before the audit-table-use steps) would prevent a future reader from wondering whether they were accidentally placed outside.

- `src/components/cmd/IngredientForm.spec093.test.tsx:241-243` — The independent-axes test seeds `caseQty: '5'` and `subUnitSize: '7'` as sentinel values so each numeric input is addressable by `screen.getByDisplayValue`. This is effective and follows an established pattern. The test comment at line 230 correctly explains the intent. One nit: the comment says "fire a case size of 20 and a sub-unit of 10" which matches the `changeText` calls but then the assertions check `caseQty` and `subUnitSize` in both positive and negative (`not.toBe`) directions — the negative direction cross-checks (`caseQty !== '10'`, `subUnitSize !== '20'`) are logically implied by the preceding positive assertions (`caseQty === '20'`, `subUnitSize === '10'`), since a string can't be both `'20'` and `'10'` simultaneously. They add noise without additional coverage. Not wrong, just redundant.

- `supabase/tests/spec093_case_qty_backfill.test.sql:296-299` — The `delete from public.order_schedule` step at line 297 mutates seed data outside the fixture scope (it is not scoped to the test fixtures' `catalog_id`/`item_id`). Because the whole test runs in a rolled-back transaction, the delete is harmless in practice. But deleting from a seed table without a `where id in (select from temp table)` scope is a potential footgun if the rollback were ever removed. A brief comment (e.g. `-- rolled back at end of txn; safe to delete broadly`) would document intent. The pattern is consistent with the reorder test file at a glance, so this is a purely advisory note.

- `src/components/cmd/IngredientForm.help-text.test.tsx:127` — The `PACK_UNIT_HELP` constant now contains a semicolon in the middle of a sentence: `"the unit each sub-unit is measured in — each, lb, oz; For abstract units..."`.
