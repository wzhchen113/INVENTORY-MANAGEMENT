## Should-fix

- `src/screens/cmd/sections/ReportsSection.tsx:123-126` — Opening a saved report does **not** reset the override state, contradicting the spec AC (line 226: "Opening a saved report resets `overrideRange` to undefined"). The implementation deliberately preserves the per-definition override across opens (comment at lines 60-62). The deviation is explained by the "mental model of open tabs" rationale, but it directly contradicts the written AC. Either update the AC to reflect the chosen behavior (preserve overrides), or add a `resetOverrides(selectedDefinitionId)` call in `onSavedReportPress` to match the spec. One of the two must be reconciled before merge so reviewers can verify the correct behavior.

- `supabase/migrations/20260511120000_report_run_cogs.sql:524-643` — The `with recursive` CTE chain (direct_ri → recursive_prep → prep_leaves → all_ri → recipe_cost → daily) is written twice in section (10): once for the `count(*)` check at line 577 and again inside the `if v_daily_count >= 2` branch at lines 582-643. Both executions hit identical base tables with identical filters. Consolidate into a single statement: `select count(*) filter (where true), coalesce(jsonb_agg(...), '[]') into v_daily_count, v_series from daily` so the planner materializes the CTE once. As written, on datasets with ≥ 2 import dates the entire recursive flatten runs twice in section (10), giving the function a minimum of 4 full recursive walks (pre-check, totals, rows, and two series walks). The inefficiency is bounded by seed-data scale today but creates a performance cliff as POS history grows.

- `supabase/migrations/20260511120000_report_run_cogs.sql:563-575` (series `daily` CTE) vs `lines:262-280` (rows `sales` CTE) — The `daily` CTE in the series section (step 10) omits the `join public.recipes r on r.id = pii.recipe_id` that is present in both `sales` CTEs (item and category branches). Because `pos_import_items.recipe_id` has `on delete cascade`, a recipe deletion also deletes its `pos_import_items` rows, so this produces identical results in practice today. However, the structural inconsistency makes the code misleading: a reader of the series CTE cannot tell from inspection alone that the missing join is safe. Add the `join public.recipes r` to `daily` for structural parity with the row aggregation, or add an inline comment explaining why the join is deliberately omitted ("cascade delete ensures referential integrity — all recipe_id values in pos_import_items reference extant recipes").

- `src/screens/cmd/sections/reports/ReportDetailFrame.tsx:228-232` — The `commitDate` function silently discards an invalid manual date entry in the chip popover without notifying the user. The comment says "the dropdown stays open so the user can try again," but `setEditingDate(null)` on line 231 closes the TextInput immediately. The user sees the date revert with no feedback — likely to be confused. The modal's `commitDateEdit` (NewReportModal.tsx:162) shows a Toast on the same path. Match the behavior: call `Toast.show({ type: 'error', text1: 'Invalid date — must be YYYY-MM-DD' })` before reverting, and update the comment to reflect that the input closes on failure (not stays open).

- `src/screens/cmd/sections/reports/templates.ts:12` — The file-level comment still says "REPORTS-2 will flip `cogs` to 'live' once `report_run_cogs` lands." That work has landed in this very spec. Update the line to past tense ("REPORTS-2 flipped `cogs` to 'live'") or remove it — leaving it as a forward-looking promise makes the file harder to audit on future reads.

## Nits

- `supabase/migrations/20260511120000_report_run_cogs.sql:118-150` — The depth-violation pre-check in step (4) walks the entire prep-recipe graph catalog-wide (not filtered to `p_store_id`) on every COGS run. Prep recipes are brand-level so the walk is correct, but the pre-check's `visited` array seeds from `rpi.prep_recipe_id` while the main aggregation's `recursive_prep.visited` seeds from `rpi.prep_recipe_id` but then appends `rp.sub_recipe_id` in the recursive arm (line 219). The two walks track slightly different node identities across graph edges. Since both physically stop at `depth < 5`, the cycle detection difference is irrelevant for the depth cap — but someone reading the pre-check to understand the cycle guard may be confused by the structural divergence. A brief inline comment noting this difference would help future readers.

- `supabase/migrations/20260511120000_report_run_cogs.sql:347-350` — The comment says "the planner can still cache temp results within the call" regarding the repeated recursive CTE. This is inaccurate: plpgsql CTEs cannot be shared across separate SQL statements; the planner materializes them independently per statement. The comment was likely intended to be aspirational. Replace with: "plpgsql does not allow CTEs to be referenced across statement boundaries — each `select … into` requires its own WITH clause."

- `src/screens/cmd/sections/ReportsSection.tsx:352` — Catalog tile footer hardcodes `range: this month`. Now that the COGS modal defaults to "Last 30d" (not "this month"), this display is misleading for the live COGS template. Either derive the displayed default from the template's configured default preset or remove the `range:` annotation from the static tile footer.

- `src/screens/cmd/sections/reports/ReportDetailFrame.tsx:83-84` — The comment "mirror NewReportModal — kept local to avoid a premature shared module before REPORTS-3 proves the shape" is a reasonable rationale, but it means `isISODate`, `toISODate`, `computePreset`, and the `PRESETS` constant are duplicated verbatim across two files. The security-auditor also flagged this (their Low finding). If REPORTS-3 brings a third copy, extract to `src/screens/cmd/sections/reports/dateHelpers.ts` at that point. Flag here so the REPORTS-3 developer knows the extraction is expected.

- `src/store/useStore.ts:1889-1894` — `db.runReport` is called with `params: def.params || {}` AND `overrideParams` separately. The store action also independently computes `mergedParams` for the optimistic row (lines 1866-1868). The merge is performed twice with the same inputs — once in the store for the optimistic row, and once in `db.runReport` for the RPC call. These two merges are structurally identical so they cannot diverge, but the duplication adds a subtle maintenance surface: if the merge strategy ever changes, it must be updated in both places. Consider centralizing the merge in the store and passing the merged result as `params` (with `overrideParams: undefined`), or document explicitly that both merge sites must stay in sync.

- `src/components/cmd/NewReportModal.tsx:159-178` — `commitDateEdit` commits a single-field date to state without checking whether the new value produces a `from > to` range. The full `from > to` guard exists in `onCreate` (lines 186-189), so no invalid range can be created, but a user who edits `from` to be after the current `to` sees no immediate feedback; the error only surfaces on CREATE. A lightweight `if (from > to)` guard in `commitDateEdit` — reverting with a Toast — would give faster feedback. Low urgency given `onCreate` catches it.
