### Should-fix

- `src/lib/auth.ts:468-470` — Stale inline comment now contradicts the fix. The comment reads "Cleanup #16 scopes the query to the current brand when brand-filtered so the table read doesn't span every tenant." That statement became false when spec 083 removed the `.eq('brand_id', brandId)` call in `db.ts`. A reader of `auth.ts` who doesn't also read the `db.ts` doc comment will believe the inference query is still brand-scoped. The architect's post-impl review labels this "acceptable" because the design explicitly said not to touch `fetchAllUsers` (read-only verify only). That scope call was correct; however a one-word amendment — changing "Cleanup #16 scopes the query …" to "Cleanup #16 previously scoped the query … (spec 083 dropped that filter — see `db.ts:fetchInvitationsForUserLookup` doc comment)" — would fix the lie without constituting a code change to `fetchAllUsers` itself. The comment is on the *call site*, not inside the function, so updating it does not refactor the function. Recommended to land in the spec 084 follow-up that already touches the symmetric `fetchBrandAdmins` path, if not now.

### Nits

- `supabase/tests/invitations_brand_id_backfill.test.sql:133-139, 210-224, 237-259` — The drift-discipline "INLINE COPY — MUST stay byte-identical to …" annotation appears on the first occurrence of each UPDATE in arm 2 (line 92-93) and arm 4 (line 157-158), but is absent from the identical SQL blocks at arm 3 (lines 133-139), arm 5 (lines 210-224), and both blocks in arm 6 (lines 237-259). The discipline comment's purpose is to alert a future editor that changing one copy requires changing all copies and the migration. Arms 3, 5, and 6 are just as much inline copies as arms 2 and 4 — a bare UPDATE block without the annotation could be silently edited without triggering the drift check. Adding a one-line comment (`-- same UPDATE #1 inline copy — see arm 2 annotation`) to each un-annotated block would close the gap. Low risk given the existing annotation in arms 2/4, but worth noting for copy discipline.

- `src/lib/db.fetchInvitationsForUserLookup.test.ts:34, 37` — `eqSpy` is a module-level `let` that is reassigned as a side-effect of `mockFrom` being called during the SUT invocation. This works correctly because `fetchInvitationsForUserLookup` calls `supabase.from('invitations')` exactly once, which triggers `makeBuilder`, which assigns `eqSpy`. The coupling between the spy capture and the invocation is implicit: if the production code ever queries a second table (or calls `from('invitations')` twice), the outer `eqSpy` variable would silently hold only the last builder's spy. A more defensive pattern — capturing the spy directly from `mockFrom.mock.results` or using a test-scoped variable in each `it` block — would make the coupling explicit. Not a bug given the current implementation; nit-level clarity.

- `src/lib/auth.fetchAllUsers.test.ts:89-95` — The `inviteRow` helper in this file initialises `brand_id: BRAND` (non-NULL) as the default, which is the opposite polarity of the bug being tested. Every test arm then overrides to `brand_id: null`. Inverting the default to `brand_id: null` (matching the actual bug fixture) would make each `it` block self-documenting without the override noise, and better aligns the helper with the template in `db.fetchInvitationsForUserLookup.test.ts:69-75` (also `brand_id: BRAND` default there, but that file's test is about confirming the row survives the query, so the override is necessary regardless). Minor; the current form is not wrong.

- `src/lib/db.ts:144` — `return (data || []) as any[]` abandons the declared return type `Promise<Array<{ email: string; profile_id: string | null; name: string; brand_id: string | null }>>` at the cast boundary, letting callers consume an `any[]` in practice. This is pre-existing (not introduced by spec 083) and out-of-scope to fix here — noted as an (out-of-scope) nit for a future tightening pass. The fix would be `return (data || []) as Array<{ email: string; profile_id: string; name: string; brand_id: string | null }>` (also correcting `profile_id: string | null` to `string` to match the NOT NULL column).
