# Code review for spec 074

## Critical

None.

## Should-fix

- `src/utils/weekWindow.ts:85` — `getWeekWindow` makes `now` a **required** argument with no default, while the architect's design (spec §"Helper signature") explicitly specified `now: Date = new Date()`, and Decision 4 states "existing `now` stays optional with default `new Date()`." The `## Files changed` block documents it as required, but the original intent was to allow wall-clock defaulting for non-test callers and interactive debugging. The only production call site always passes `now`, so there is no runtime breakage today — but the deviation removes a useful affordance for future callers and diverges from the signed-off spec. Either add the default (`now: Date = new Date()`) or add a comment in the function doc explicitly justifying the departure from Decision 4.

- `src/lib/cmdSelectors.unconfirmedPoWindow.test.ts:158–163` — The "out-of-window row excluded" test is vacuously true. The test asserts that the id `store-1:po:vendor-v:2026-05-20` does not appear in the output for `now = 2026-05-27`. This can never appear regardless of the window filter, because `isoDateRange(mondayStart, nextMondayStart)` only enumerates dates in the current week (Mon 2026-05-25 through Sun 2026-05-31) by construction — `2026-05-20` is structurally impossible in the output. The test would pass even if `pastISOsInWindow = weekISOs` (no `< todayISOInTz` filter at all). A real out-of-window test requires a scenario where a date _would_ have been emitted without the filter: e.g., add a second fixture vendor scheduled on Wednesdays with no submission on `2026-05-20` (prior week), run with `now = 2026-05-20T14:00:00Z` so that date IS in the current-week range, then change `now = 2026-05-27T14:00:00Z` and assert it no longer appears — or simply assert that no emitted id has a date earlier than the week-start ISO.

## Nits

- `src/utils/weekWindow.ts` (header comment, lines 1–29) — The implementation diverges from the architect's `## Frontend design` helper signatures in two ways: (1) `WeekWindow` exports `{ mondayStart: Date; nextMondayStart: Date }` instead of `{ weekStartISO: string; todayISO: string }`; (2) `isoDateRange` takes `Date` objects rather than ISO strings. The header comment explains the UTC-anchoring rationale but does not explicitly flag these as departures from the signed-off spec. A reader cross-referencing `## Frontend design` will see mismatched types. A single sentence "(Architect's spec §"Helper signature" specified ISO strings; this implementation uses UTC-anchored Date objects — see rationale in header above)" in the JSDoc of either export would close the gap.

- `src/utils/weekWindow.test.ts:21–26` — The "Monday on a Monday" test only asserts `isoDateRange(...)[0]` equals `'2026-05-25'`, not the length or the last element. The parallel Wednesday test (lines 12–18) asserts the full 7-element array. For consistency and to guard against an off-by-one in the `nextMondayStart` computation on a Monday, asserting the full week array (or at minimum both `[0]` and `[6]` plus `.length`) would match the coverage depth of the other day-of-week tests.

- `src/lib/cmdSelectors.unconfirmedPoWindow.test.ts:121` — The test description says "Sunday night just before midnight" and the comment in the test body says "still in current work-week," which is correct. However the spec AC says `now = 2026-05-31T23:00:00 (a Sunday)` while the test uses `2026-06-01T03:59:00Z` (= 2026-05-31 23:59 EDT). The times are both Sunday night in NY and both pass, but the comment would be clearer if it noted why the UTC timestamp is `T03:59Z` not `T23:00Z` (i.e. the Z-suffix is UTC, not local time) — the conversion tripped the spec author and may trip a future reader adjusting this fixture.

## Resolution (post-review FE fix-pass — main Claude)

- Should-fix #1 — **fixed.** `getWeekWindow(tz, now)` → `getWeekWindow(tz, now: Date = new Date())`. Architect Decision 4 satisfied; production call site unchanged.
- Should-fix #2 — **fixed.** The vacuously-true literal-id check at `cmdSelectors.unconfirmedPoWindow.test.ts:158` was refactored into a structural invariant: assert every emitted id's ISO suffix is ≥ `weekStartISO` (which WOULD fail if `pastISOsInWindow` were accidentally widened to include prior weeks). The original literal `not.toContain(...)` line is retained as a quick belt-and-suspenders check. test-engineer's separately-added today-exclusion test further closes the gap on the other direction.
- Nit #1 — **partially addressed via JSDoc.** Added a paragraph to the `getWeekWindow` JSDoc explicitly noting the return-shape divergence from the architect's original ISO-string sketch and the rationale (Date arithmetic is easier than repeated parse/format at the one consumer boundary).
- Nit #2 — **fixed.** The "Monday on a Monday" test now asserts the full week shape (`toEqual(['2026-05-25', ..., '2026-05-31'])`), mirroring the Wednesday test and guarding against a `nextMondayStart` off-by-one on the boundary day.
- Nit #3 — **fixed.** The "Sunday night just before midnight" test comment now spells out the EDT→UTC offset explicitly so a future reader adjusting the fixture doesn't trip on the Z-suffix.
