## Code review for spec 020

### Critical

- `src/lib/db.ts:608-624` — `fetchRecentEodDates` does not deduplicate dates. After spec 020 lands, a store that has two vendors both submitted on the same day will have two `eod_submissions` rows for that date. The PostgREST query `.select('date').limit(2)` returns two rows with the same date string. `seedVarianceDates` in `NewReportModal.tsx:70` uses `dates[1]` as `from` and `dates[0]` as `to`; when both equal the same date, `report_run_variance` raises `errcode = '22023'` ("from == to") before even reaching the P0002 anchor gate. The NewReportModal variance template becomes broken for any store that has submitted two vendors on today's date. Fix: add `.order('date', { ascending: false })` (already present) and deduplicate by mapping to unique dates before returning — e.g., `return [...new Set((data || []).map((r: any) => r.date as string))].slice(0, limit);` — or switch the query to `select=date&order=date.desc` with a DISTINCT modifier via an RPC or a `?select=date` + client-side dedup.

- `src/store/useStore.ts:1374-1405` — The Q6 vendor-scoped `current_stock` guard is inverted for null-vendor items. Line 1375: `!subVendorId || !item?.vendorId || item.vendorId === subVendorId`. The `!item?.vendorId` branch evaluates to `true` when an inventory item has no `vendorId`, so null-vendor items are treated as matching and their `currentStock`/`eodRemaining` are updated in local state AND `db.adjustItemStock` is called (line 1400) — which writes `current_stock` to the DB without any vendor filter. The RPC correctly blocks this with `WHERE id = ... AND vendor_id = p_vendor_id`, but the admin-JWT path (via `adjustItemStock`) does not. Q6 says escape-hatch items should NOT have their inventory mutated. The correct guard is `item.vendorId === subVendorId` only (removing the `!item?.vendorId` short-circuit). Items with null vendorId should fall through to the audit row only.

- `src/screens/EODCountScreen.tsx:514-525` and `src/screens/EODCountScreen.tsx:659-669` — Legacy screen constructs `Omit<EODSubmission, 'id'>` objects without `vendorId`. Because `EODSubmission.vendorId: string` is now non-optional (spec 020 change in `src/types/index.ts:241`), and because `persistToCloud` is typed as `Parameters<typeof submitEODCount>[0]`, these are TypeScript compile errors. TypeScript strict mode will reject these lines. The spec (scrutiny item 12) acknowledges this as a known issue but does not resolve it. Per CLAUDE.md, legacy screens still ship — this spec ships with a broken TypeScript build. User decision needed: either (a) add `vendorId: ''` as a temporary stub so the legacy path compiles (with a comment that null vendor_id is now forbidden by the DB constraint, so this path will produce a DB error if actually executed), or (b) gate the legacy screen behind a compile-time `@ts-expect-error` with a TODO. Leaving it as broken TS errors is not acceptable to ship.

### Should-fix

- `src/screens/cmd/sections/EODCountSection.tsx:491-512` — In `onSubmit`, the vendor's draft state (`setCaseCountsByVendor`, `setUnitCountsByVendor`, `setNotesByVendor`) and `editingVendorIds` are cleared **before** `await submitEODCount(submission)` resolves. If the cloud write fails (line 508: "Saved locally only"), the user's raw input is already gone. Because the optimistic `eodSubmissions` slice was updated first (`submitEOD(submission)` at line 491), the vendor immediately renders as LOCKED — the user would have to tap EDIT to recover values from the optimistic snapshot, and only items that were part of the original `filteredItems` would be recoverable. Typed-but-unsaved escape-hatch items added via `+ COUNT` are silently lost if the cloud call fails. Move the `setCaseCountsByVendor`/`setUnitCountsByVendor`/`setNotesByVendor` clears to inside the `try` block after `await submitEODCount` succeeds (line 506). `setEditingVendorIds` should also move there.

- `supabase/migrations/20260514120010_staff_submit_eod_v2.sql:91-92` — The upsert inserts `null` for `submitted_by`. The column comment on `eod_submissions` (from init_schema) says `submitted_by` is a FK to `profiles(id)`. Inserting `null` is allowed by the column definition, but the 7-arg RPC accepts `p_submitted_by text` (a display name, not a UUID) and never uses it to populate the FK. The audit row correctly uses `p_submitted_by` for the `detail` field, but the submission row's `submitted_by` FK stays null for staff-app submissions. This is inherited from v1 and probably intentional (staff-app users have no profiles row), but the comment "null" should be made explicit — e.g., a one-line note explaining that `submitted_by` is null for staff-app callers.

- `src/store/useStore.ts:1409-1412` — Audit detail format diverges between the admin-JWT path (store action) and the staff-app path (RPC). The store emits `"Remaining count submitted · vendor: BJs"` or `"Count updated · vendor: BJs"`. The RPC emits `"<p_submitted_by> · vendor: BJs"`. The spec AC says "vendor identifiable in audit detail" (met by both), but the architect's design note says both paths should produce a consistent suffix. This makes audit log parsing and searching inconsistent. Suggest aligning: both should use the action name as the base (e.g., `"EOD entry · vendor: <name>"`) or document the intentional difference.

### Nits

- `supabase/migrations/20260514120000_eod_submissions_vendor_id.sql:103-110` — Four separate `DROP CONSTRAINT IF EXISTS` / `DROP INDEX IF EXISTS` statements for the old unique constraint, covering two possible historical naming patterns. This is defensive and correct, but a comment explaining why both names could exist (prod used one migration path, local used another) would help future readers understand why four statements are needed instead of one.

- `supabase/migrations/20260514120010_staff_submit_eod_v2.sql:79-85` — The `client_uuid` idempotency check returns early if the UUID matches, regardless of whether the existing submission has a different `(store_id, date, vendor_id)`. The comment at line 73 acknowledges this. This is consistent with v1 behavior, but the edge case (a retry from the staff app that somehow uses the same `client_uuid` for a different vendor) silently returns the wrong submission id. The comment is there, so this is a documentation nit: clarify whether this is a "won't fix" or "acceptable because client_uuid is cryptographically random."

- `src/screens/cmd/sections/EODCountSection.tsx:286-294` — `currentVendorSubmission` is a `React.useMemo` that calls `.find()` on `eodSubmissions`. The dependency array includes `eodSubmissions` as a whole. Since `eodSubmissions` is the full Zustand slice and is replaced on every realtime tick, this memo recomputes on every unrelated EOD update. Not a bug, but consider subscribing only to submissions for the current store + date if realtime volume becomes noticeable.

- `src/screens/cmd/sections/EODCountSection.tsx:769` — The button text `'UPDATE COUNT'` vs `'SUBMIT COUNT'` correctly reflects EDIT mode vs fresh submit. No issue, but the spec scrutiny item 8 asks to verify this — confirmed: it is driven by `isCurrentVendorEditing` and is correct.

- `src/lib/db.ts:426` — `fetchTodaysEODForStores` is declared as returning `Promise<any[]>` while `fetchEodSubmissionsForStores` (a sibling function, same shape) returns `Promise<EODSubmission[]>`. With the new `vendorId` field added to both, the weaker `any[]` return type on `fetchTodaysEODForStores` means callers won't get compile-time checking on the returned objects. (Out-of-scope pre-existing inconsistency — noting for follow-up.)

- `src/screens/cmd/sections/EODCountSection.tsx:1455` — `VarianceLogTab` uses `item.parLevel` as the "expected" value for the client-side variance display. The comment "(par as proxy when no expected stock signal)" is correct, but the server-side `report_run_variance` uses the prior EOD count as "expected" — these are fundamentally different baselines. The tab's description "Counted vs expected per item" slightly misrepresents what is shown; "Counted vs par level" would be more accurate.
