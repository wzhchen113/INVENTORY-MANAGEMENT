### Should-fix

- `supabase/tests/report_run_custom.test.sql` (entire file) — **The spec-required jest Track-1 test (`src/components/cmd/NewReportModal.test.tsx`) was not delivered.** Spec §Tests / Track 1 (jest) pinned five concrete assertions: SQL textarea visible, date-range chips absent, CREATE with empty SQL toasts "SQL required" without calling `addReportDefinition`, CREATE with non-empty SQL calls `addReportDefinition` with `params: { sql }` and no `range`/`from`/`to`/`by` keys. The pgTAP suite passes all 13 arms and covers the backend, but the modal's `isCustom` branching logic has zero automated coverage on the frontend side. Coordinate with the test-engineer (who owns the jest-track pass/fail verdict) before marking SHIP_READY.

- `supabase/tests/report_run_custom.test.sql` — **Spec-pinned pgTAP arm 7 (RLS cross-store enforcement) is missing.** The spec (§Tests, arm 7) requires asserting that an admin who is a member of Frederick calling `SELECT * FROM public.inventory_items WHERE store_id = '<charles-id>'::uuid` returns 0 rows — anchoring the "RLS, not lexical injection" posture from §A2.4. The 13-arm plan instead adds two DML-rejection arms (arms 6 and 7 both test 42601-class parse errors for bare INSERT and bare CREATE TABLE) where the spec budgeted one timeout arm (arm 6, acknowledged as untestable) and one RLS-enforcement arm (arm 7). The DML rejections are well-covered, but the RLS cross-store property — the load-bearing claim of the spec's security posture — is not exercised. The test comment in arm 8 calls itself "LOAD-BEARING" but arm 7's equivalent is missing. A happy-path SELECT against `public.stores` (arm 12) is not a substitute.

- `supabase/migrations/20260515130000_report_run_custom.sql:245-246` — **`when others` catch-all re-raises with `errcode = 'P0001'`, which collides with `assert_not_last_of_role` and the variance `P0001` raises already in the codebase.** `P0001` is the default `raise_exception` SQLSTATE — any caller that pattern-matches on this code (e.g. a future `db.ts` branch, or a pgTAP `throws_ok` on the parent dispatcher) cannot distinguish "custom SQL run failed" from "cannot delete the last super_admin." A custom SQLSTATE like `'P0501'` (within the user-defined `P05xx` range, which Postgres reserves for application use) would be unambiguous without any Postgres compatibility risk. As-is, the `db.ts` allowlist branch catches the right message string so there's no current breakage, but the SQLSTATE choice is a readability and future-maintainability hazard.

- `src/components/cmd/NewReportModal.tsx:467` — **Hint text says "5s timeout" but the actual enforced budget is 8s.** The migration header (line 17-28) explicitly documents that `SET LOCAL statement_timeout = '5s'` does NOT fire inside a plpgsql function body; the operative wall-clock budget is the `authenticated` role's connection-level 8s default. The UI hint `'SELECT only · public.* tables · 5s timeout · max 1000 rows · RLS-filtered to your stores'` therefore misrepresents the SLA to the user. Change "5s timeout" to "8s timeout" to match the documented reality, or to "5s timeout*" with a tooltip if precision matters. Using the wrong number can lead users to believe a 6-second query is within budget when it is not.

- `src/screens/cmd/sections/reports/ReportDetailFrame.tsx` (entire `ResultBody` component) — `output._row_count` is declared in `ReportRunOutput` and populated by the runner, but is never consumed in the UI. The spec §"Envelope shape" describes it as "Lets the frame show '1000 rows' / '47 rows' in the table header without a second `.length` read." The frame uses `rows.length` implicitly by iterating, but never renders a row-count label. The field is emitted from the backend, typed on the frontend, and then silently dropped. Not a bug (the table still renders all rows), but the field's stated purpose ("lets the FE render") is unmet. Consider adding a small "N rows" label above the `ResultTable` for custom reports, or document that `_row_count` is reserved for a follow-up.
