## Security audit for spec 025

Scope: deletion of legacy nav/screens/data layer, new `UsersSection.tsx` + `InviteUserDrawer.tsx`, new `sendPasswordReset(email)` helper in `src/lib/auth.ts`, ReorderSection CSV/PDF export, tsconfig exclusion additions, and the new `typecheck-base` CI job. Auditor was read-only on code; only mutating action was `npm audit --audit-level=high`.

### Critical (BLOCKS merge)

None.

### High (must fix before deploy)

None.

### Medium

- `src/screens/cmd/sections/UsersSection.tsx:62-66`, `src/lib/cmdSelectors.ts:1081-1086` ā€” **`Users & access` sidebar entry is visible to all admins, but the underlying RLS policy only gives non-super-admins their own profile row**. A non-super-admin / non-master `admin`-role user will see a sidebar entry labelled "Users & access", click it, and get a list containing only themselves. That's not a data leak (RLS does the right thing ā€” see `supabase/migrations/20260405000759_init_schema.sql:261-262` "Own profile" policy and `supabase/migrations/20260509000000_multi_brand_schema_rls.sql:981-988` super-admin additive policy), but it is a misleading UX surface: the section labels itself "admin-global" while delivering a single-row view, and it gives a non-master admin a per-row "DELETE" / "RESET PW" button against their own account. Recommend either: (a) gate the sidebar group on `useIsMaster()` (master + super_admin) at the selector layer, the way `Tenancy/Brands` is gated; or (b) add a defensive role gate in the `UsersSection` body so admins get an explicit "you don't have access to this view" panel instead of a one-row list of themselves. Fix is one line. Pure UX/affordance issue ā€” server side is sound.

- `src/components/cmd/InviteUserDrawer.tsx:91-108`, `src/lib/auth.ts:151-203` ā€” **Invitation flow accepts client-supplied `brandId` and `storeIds` without server-side cross-checking that the caller can see that brand**. The `invitations` table INSERT policy (`supabase/migrations/20260424211733_security_fixes.sql:46-48`) only checks `app_metadata.role IN ('admin','master')` ā€” it does not verify the inserting admin can see the `brand_id` they're attaching to the invitation, nor that the inserting admin can see every `store_id` in `store_ids`. A multi-brand admin (or a single-brand admin whose JWT carries `role='admin'`) could craft a direct PostgREST POST to `invitations` with arbitrary `brand_id` and `store_ids`. The `user_stores_brand_match` trigger in `supabase/migrations/20260509000000_multi_brand_schema_rls.sql` catches the cross-brand store assignment at registration time, but the invitation row itself lands successfully ā€” which (1) sends an invite email, (2) lets the invitee register and create a profile, and (3) only fails at the `user_stores.insert` step inside `registerInvitedUser` (where the trigger fires). Pre-existing (the policy shape predates spec 025), but spec 025 makes this the primary surface for admin-invokable invites; recommend tightening the invitations INSERT policy in a follow-up to also enforce `auth_can_see_brand(brand_id)` and a constraint that every `store_id` in `store_ids` belongs to that brand.

- `src/lib/auth.ts:177-186` ā€” **`inviteUser` writes `opts.email`, `opts.name`, `opts.storeIds`, `opts.brandId` straight to the `invitations` table; `send-invite-email` then interpolates them into an HTML email body without escaping** (`supabase/functions/send-invite-email/index.ts:70`). A malicious admin (or one whose session was stolen) could craft `name = '<script>...</script>'` and the email content would render that markup in any web-mail client that doesn't strip script tags. The role/store-IDs fields aren't surfaced into HTML, but `name` and `email` are, and `storeNames` is built client-side and is also interpolated directly. Pre-existing (the edge function predates spec 025), but spec 025 promotes the invite flow from a niche legacy screen to a sidebar-visible admin tool; recommend HTML-escaping interpolations inside `send-invite-email/index.ts:70`. Same vector applies to `send-welcome-email/index.ts:71`.

- `src/lib/auth.ts:408-416` ā€” **`sendPasswordReset` is callable by any authenticated user against any email address**. `supabase.auth.resetPasswordForEmail(email)` is the unauthenticated GoTrue endpoint ā€” it doesn't actually require the caller to be admin. The UsersSection enforces admin-only role gates client-side (`UsersSection.tsx:286-288`), but those gates are advisory: the function is reachable from any browser-side code path that imports `sendPasswordReset` (today none beyond UsersSection ā€” but no server check exists). Supabase's project-level rate limit at `supabase/config.toml:192` (`email_sent = 2/hr`) is the only abuse limiter. Combined with the existence of `EXPO_PUBLIC_SUPABASE_URL` being public, anyone with the URL can hit the GoTrue endpoint directly without any admin context anyway ā€” so a deliberate password-reset spam attack doesn't strictly need this helper. The spec correctly notes this in Ā§2.B ("Server-side this is gated by Supabase's GoTrue policy") ā€” calling out as a Medium because the helper's naming + admin-UI placement could mislead future maintainers into thinking it carries an admin-side authorization check. Mitigation: a one-line JSDoc note that this is fundamentally a "trigger the public forgot-password mailer for an arbitrary email" call, plus a comment near the call site documenting the rate-limit behavior.

- `supabase/migrations/20260424211733_security_fixes.sql:46-57` ā€” **`invitations` table RLS still hard-codes `['admin','master']` and does not include `super_admin`**. A super-admin promoted by `20260509000000_multi_brand_schema_rls.sql` (e.g., `wzhchen113@gmail.com`) has `app_metadata.role = 'super_admin'`, which fails the policy's IN-list check. Result: when a super-admin uses the new `InviteUserDrawer` (which the spec explicitly extends to super_admin via `isMaster = role === 'master' || role === 'super_admin'`), the invitation INSERT will fail with an RLS rejection. Pre-existing (same gap exists in the legacy `UsersScreen`), but spec 025's new section foregrounds the bug and the spec design explicitly says super-admins are treated as master in the UI. Recommend updating the four `invitations` policies to include `super_admin` in the IN-list (mirroring the pattern in `delete-user/index.ts:19` which already does this). Not a *new* security gap; a *new* product-correctness regression that lands with this PR.

### Low

- `src/lib/auth.ts:115` ā€” `process.env.EXPO_PUBLIC_SUPABASE_URL` is fetched at runtime inside `callEdgeFunction`; if the env var is undefined, the `fetch` URL becomes `undefined/functions/v1/...`. The function silently swallows errors via the empty `catch {}` (line 124), so the calling flow (`inviteUser`) reports success even though the email never went out. Spec 025 doesn't change this code, but it's worth flagging because the new UsersSection is the primary path that exercises `send-invite-email`, and a missing env var would surface as "invitation didn't send" with no client-side error. Pre-existing.

- `src/screens/cmd/sections/UsersSection.tsx:118-127` ā€” **Self-delete UX path is unreachable**. UI offers self-delete with "signing out, redirect to login" UX, but `supabase/functions/delete-user/index.ts:59-64` rejects `userId === gate.userId` with `400 cannot delete self`. The UI's `handleConfirmDelete` flow at line 113-131 routes through `deleteProfile` ā†’ `deleteUser` edge call ā†’ 400 ā†’ `ok` is `false` ā†’ modal stays open ā†’ no logout, no redirect. Not a security issue per se (server-side is the safer behavior), but the UX/spec mismatch is worth flagging since the spec AC24 explicitly says "If the deleted user is self, sign out + redirect to login" ā€” that path will never execute. Code-reviewer/test-engineer territory.

- `src/screens/cmd/sections/ReorderSection.tsx:383-401` ā€” CSV export writes `vendor.vendorName`, `item.itemName`, and `item.unit` directly into the CSV body via `Papa.unparse`. PapaParse handles quote-escaping correctly. **CSV injection** (Excel formula execution when a cell starts with `=`, `+`, `-`, `@`) is theoretical here: the data source is `report_reorder_list` RPC output, which is sourced from `vendors.name` / `inventory_items.name` ā€” admin-written fields, not untrusted user input. Low risk; flagging only because a brand-name field starting with `=` would still execute as a formula in Excel. Mitigation if desired: prefix `'` to any cell whose first char is `=+-@`. Not blocking for admin-only data.

- `src/screens/cmd/sections/ReorderSection.tsx:471-558` ā€” `jsPDF` + `jspdf-autotable` are dynamic-imported per spec AC5. Confirmed via `npm ls`: both bundle in `node_modules` (no runtime CDN load). PDF rendering is local-only; no SSRF surface.

- `src/lib/cmdSelectors.ts:1078-1086` ā€” New `Admin > Users` sidebar group is added without a feature-flag guard. The deletion of `EXPO_PUBLIC_NEW_UI` is correctly handled (no auth implication), but this expanded sidebar default ships to all admins as soon as the build deploys. Not a security finding ā€” flagged here only because the sidebar override system at spec 008 stores per-user sidebar layouts in `profiles.sidebar_layout` and the new `id: 'Users'` is the load-bearing key; if a future migration ever renames it, every user with a customized sidebar layout would lose the entry. Architectural concern, not security.

- `tsconfig.json:11-16` exclude of `scripts/**` and `supabase/functions/**` is correct: edge functions run under Deno (separate type system), `scripts/` are one-off ts-node smokes that don't ship to the runtime. No new attack surface from the exclusion ā€” the excluded files were never part of the bundle. Type-system coverage of edge functions is owned by the Deno LSP / `supabase functions deploy` typecheck; that hasn't regressed.

### Dependencies

`npm audit --audit-level=high` reports **1 high**, 5 moderate, 5 low (no critical):

- **High**: `@xmldom/xmldom <=0.8.12` ā€” uncontrolled recursion in XML serialization (DoS) + XML node injection via doctype / processing-instruction / comment serialization. Transitive via `expo` ā†’ `@expo/cli` ā†’ `@expo/plist` (plist generation for native build only). Path: `expo@54.0.33 ā†’ @expo/cli@54.0.23 ā†’ @expo/plist@0.4.8 ā†’ @xmldom/xmldom@0.8.12`. **Not reachable from runtime app code** ā€” only fires during EAS native builds (plist + Xcode project generation). Build-server threat surface only; recommend `npm audit fix` in a follow-up since it appears non-breaking per audit output. Pre-existing across the codebase, not introduced by spec 025.
- 5 moderate: `dompurify <=3.3.3` (multiple sanitization bypasses), `postcss <8.5.10` (XSS via `</style>`), `@expo/metro-config`, `@expo/cli`, `expo` transitive on the postcss issue. All build-tooling / dev-server surface; not exposed in shipped bundle.
- 5 low: `@tootallnate/once`, `http-proxy-agent`, `jsdom`, `jest-environment-jsdom`, `jest-expo` ā€” all test-only dependencies; not in production bundle.

No package.json changes in this PR introduced new vulnerabilities. The single net-new dependency surface (`papaparse`, `jspdf`, `jspdf-autotable`) was already present in the legacy `OrderReportScreen` and is unchanged in version. Existing low/moderate counts are unchanged.

### Summary

No Critical or High findings. The five Medium findings are all spec-design + pre-existing-policy issues that spec 025 makes more visible by foregrounding the user-management surface; none of them are exploit chains landing in this PR. The Low findings are observations for follow-up cleanup. **Recommended for ship**, with two follow-up tickets suggested:

1. Tighten `invitations` RLS to (a) include `super_admin` and (b) cross-check `brand_id` + `store_ids` against the caller's visible brand/stores.
2. HTML-escape interpolated user input in `send-invite-email/index.ts:70` and `send-welcome-email/index.ts:71`.

Neither is a spec-025 regression ā€” both predate the spec ā€” but both become more reachable when the user-management surface is added to the canonical Cmd UI sidebar.
