### Should-fix

- `supabase/tests/compute_menu_capacity.test.sql:30,42` — **`plan(16)` count is correct, but the perf test listed in the header as item (12) ("Perf: < 100ms on the seed") is never implemented.** The header comment at line 30 lists "(12) Perf: < 100ms on the seed" as one of the 12 cases covered, but no `clock_timestamp()` assertion exists in the file body. The 16-assertion count is accurate (the fixture-resolution `isnt` at line 70 fills the count), so pgTAP will not fail on plan mismatch — but the spec's AC §C explicitly requires a `< 100 ms p95` pgTAP probe ("pgTAP probe wrapping the RPC in `clock_timestamp()` deltas"), and the Verification section claims it passes. The perf acceptance criterion from the spec is simply absent. Fix: add a `clock_timestamp()` delta assertion against the seed store (matching the shape described in the test plan: assert `extract(milliseconds from ...) < 100`). Note: if adding this assertion, increment `select plan(...)` by 1. Deferred to test-engineer for formal coverage gap tracking, but calling it out here because it's a spec-AC miss.

- `src/components/cmd/MenuCapacityBadge.tsx:88,138-139` — **`borderColor` is a dead variable.** Declared as `let borderColor: string | undefined` at line 88 but never assigned in any branch. The pill `style` at line 138-139 uses `borderWidth: borderColor ? 1 : 0` (always `0`) and `borderColor` (always `undefined`). If a future state (e.g., a "critical" tier with a red border) was intended, there is no code to set it. If borders are not needed, the two lines should be removed. As-is, TypeScript strict mode will not flag this because `undefined` is a valid value for `borderColor: string | undefined`, so the unused variable slips through. Fix: remove `let borderColor: string | undefined` and the two `borderColor`-referencing lines from the pill style, or assign a value if a bordered state is actually intended.

- `supabase/tests/compute_menu_capacity.test.sql:402-421` — **The `truncated` column is never asserted in pgTAP.** The cycle test at (8) correctly asserts `makeable_qty = 4` and the comment claims `truncated=true` is expected, but there is no `select is((select truncated from _cap where ...), true, ...)` assertion. Furthermore the comment's reasoning about WHY `truncated` fires is incorrect: for a 2-node cycle (`prep_x ↔ prep_y`), the visited-array terminates recursion at depth 2, long before the `depth = 5` predicate in `truncated_recipes` can fire. A 2-node cycle therefore returns `truncated = false`, not `truncated = true`. The `truncated` column — the primary defense-in-depth signal the spec added for cycle visibility — goes unverified. Fix: (a) add a `select is((select truncated from _cap ...), false, ...)` assertion to lock the actual behavior for the 2-node cycle, and add a separate assertion with a truly deep (depth ≥ 5) chain fixture to verify `truncated = true` when the depth cap is hit. Increment `plan(...)` accordingly. Also correct the misleading comment at lines 405-415.

### Nits

- `supabase/migrations/20260524000000_compute_menu_capacity_rpc.sql:218-220` — The `recipe_lines` `GROUP BY` includes the joined columns `ii.current_stock, ii.par_level, ci.name, ci.unit` as explicit group keys. These are functionally dependent on `a.catalog_id` (via the `(store_id, catalog_id)` unique constraint on `inventory_items` and the PK on `catalog_ingredients`), so adding them to the GROUP BY is safe but redundant compared to the canonical pattern in `report_run_variance_multivendor` which aggregates in `all_ri` and keeps group keys minimal. Minor structural divergence from the canonical but not a bug.

- `supabase/tests/compute_menu_capacity.test.sql:167-171` — `LOW5` recipe is mentioned in the fixture comment ("Recipe LOW5: direct cat_a + cat_b + cat_c + cat_zero + extra-low (5 ingredients total)") at line 170 but is never created in the INSERT at line 177. Low severity — the counter logic is correct and tested, just not at the stated scale. Deferred to test-engineer.

- `src/screens/cmd/sections/MenuImpactSection.tsx:141` — `hasRecipe: cap ? cap.hasRecipe : true` defaults to `true` when the RPC hasn't replied yet. This is the correct defensive default (rows stay in the normal sort order until the RPC confirms they have no BOM), and the design's rationale is sound. Worth a brief inline comment explaining why `true` is the correct default rather than `false`; without a comment, the next reader may suspect a logic inversion.

- `src/components/cmd/MenuCapacityBadge.tsx:66` — `if (qty === null || qty === undefined)` — the TypeScript type for `makeableQty` is `number | null`; `undefined` is not in the union. Since this project targets TypeScript strict, the `=== undefined` check is unreachable. Fine to keep for defensive JS runtime safety, but a brief comment would clarify the intent ("JS runtime guard — TS type is `number | null`").

- `src/screens/cmd/sections/__tests__/MenuImpactSection.test.tsx:128-129` — `const expectedAria = ...` is computed but immediately voided at line 144 (`void expectedAria`). - `src/screens/cmd/sections/__tests__/MenuImpactSection.test.tsx:388` — `void within;` at the end of the `emits the unit-mismatch indicator` test. `within` is imported at line 110 but only used here as `void within` to satisfy a "declared but not used" lint.
