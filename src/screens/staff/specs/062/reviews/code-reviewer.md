# Code review for spec 062

## Critical

- `src/navigation/RootStack.tsx:107,109` — `authState.toast` is set to a raw i18n key string (`'auth.error.notStaff'`, `'auth.error.noStores'`) rather than a translated message. More importantly, the `toast` field on `AuthState` is written in three places (`SignIn.tsx:98`, `RootStack.tsx:107,109`, `EODCount.tsx:271`) but **never read anywhere** to fire an actual toast. The cold-start gate failure path (`restoreSession` finds a non-staff or no-stores user) sets `authState.toast` but the `SignIn` screen never inspects it and never calls `Toast.show`. Result: a user whose role was demoted since last launch gets silently redirected to `SignIn` with zero explanation — violating spec §B3 ("show an error toast") and the state-machine contract in §2. Fix: in `RootStack.tsx`, call `Toast.show` before or alongside `setAuthState({ kind: 'signed-out' })` when the gate fails; don't rely on the `toast` field at all. Also fix `RootStack.tsx:107,109` to pass `t('auth.error.notStaff')` / `t('auth.error.noStores')` if the field is retained.

## Should-fix

- `src/navigation/RootStack.tsx` (whole file) and `src/screens/SignIn.tsx:24-59` — The role + `user_stores` gate logic is fully duplicated: `runGate(userId)` in `SignIn.tsx` and `restoreSession()` in `RootStack.tsx` perform the identical two-query sequence independently. Any future change to the gate (e.g. checking a second role flag, or a different error message) must be applied in two places. Extract into a single `checkAuthGate(userId)` function in `src/lib/` and call it from both sites.

- `App.tsx:25-26` — `migrateQueueIfNeeded()` internally calls `hydrateQueue()` and throws away the result (see `eodQueue.ts:101-106`), then `App.tsx` immediately calls `hydrateQueue()` again. This causes two sequential AsyncStorage reads of the same key on every cold start. Either have `migrateQueueIfNeeded()` return the already-validated array so `App.tsx` can use it, or restructure so `hydrateQueue()` is called once. As-is it is wasteful and confusing.

- `src/hooks/useEodSubmit.ts:179-185` — In the drain loop, a non-forbidden non-network error (5xx / malformed response) bumps attempts and continues to the next item. The spec state machine (§3) categorizes this path as `failed` for the interactive submit, but the drain equivalent is ambiguous. More concretely: the `bumpEodAttempts` call here passes the raw `err.message` as `lastError`, which may be lengthy or contain server internals. This path also has no corresponding test case in `useEodSubmit.test.ts` — the hook test covers success/success-replay/forbidden/network/failed for the *interactive* submit, but does not exercise a 5xx during the *drain* loop. Add a drain-5xx test so the bump-and-continue behavior is explicitly verified.

- `src/screens/EODCount.tsx:297-305` — When the user presses Submit with all inputs blank (`entries.length === 0`), the code fires `Toast.show` with `text2: 'No counts entered'` — that string is hardcoded in English, bypassing the i18n channel. All user-facing strings must go through `t()` (spec §B8). Add a key like `eod.toast.noCountsEntered` to `en.json` and use it here.

- `src/i18n/i18n.test.ts:35-83` — The required-keys list in the coverage test is missing `auth.error.generic`, which is used in `SignIn.tsx:119`. If that key were accidentally deleted from `en.json`, the test would not catch it. Add `'auth.error.generic'` to the list.

- `src/i18n/i18n.test.ts:61` — `chrome.queue.queuedRow` is in the required-keys list but is not referenced anywhere in production code. The queued-outcome toast in `EODCount.tsx:346` uses `eod.toast.queued` instead. Remove `chrome.queue.queuedRow` from the required list or add a production callsite — a key in the required list that is not called in code is misleading and the test would pass even if the callsite never reached it.

- `src/components/Button.tsx:89` — Inline hex-adjacent color literal `rgba(30,136,229,0.08)` for the secondary pressed state. The project convention (CLAUDE.md) is to use theme tokens rather than hardcoded color values. `colors.primaryPressed` at `#1565c0` doesn't work here, but a new `primaryPressedLight` token (or similar) should be added to `src/theme.ts` rather than embedding the raw value in the component.

- `.claude/launch.json:5` — Configuration name is `"expo-web"`, but spec §10 prescribes `"expo-web-staff"` to avoid collision with imr-inventory's launch config (which is also likely named `"expo-web"`). When both repos are open in the same IDE session, this causes the wrong config to be activated. Rename to `"expo-web-staff"`.

## Nits

- `src/hooks/useEodSubmit.ts:150-155` — During drain, a `conflict: true` (idempotent replay) result fires a per-item toast "Already submitted — your counts match..." for every replayed item. In a drain cycle with five replayed items this fires five identical toasts. The spec says `"All counts synced"` should fire once per drain cycle at the end. Consider suppressing the per-item replay toast during drain (it fires in the interactive submit path where it makes sense, but not in the drain loop).

- `src/lib/eodQueue.ts:96-107` (`migrateQueueIfNeeded`) — The function calls `hydrateQueue()` solely to trigger corrupt-payload backup side effects, then discards the return value. The comment says "the returned array is discarded because the App mount effect calls hydrateQueue() right after" — but this makes the function's intent opaque. The double-read concern (noted in Should-fix above) makes this more confusing. If migration truly has no v1 transforms, consider making `migrateQueueIfNeeded` a no-op stub with a comment explaining it activates only for v2+, and letting `hydrateQueue()` continue to handle corrupt-payload backup on its own.

- `src/navigation/RootStack.tsx:145-177` — `let content: React.ReactElement` is assigned in a series of `if/else` branches and then returned as `<NavigationContainer>{content}</NavigationContainer>`. This works, but TypeScript's exhaustive analysis doesn't enforce that every branch is covered. A helper that returns the navigator JSX directly from each branch (or uses an early-return pattern) would make the exhaustion explicit and is idiomatic for conditional render.

- `src/screens/EODCount.tsx:82-112` (`fetchVendorsForToday`) — The `seen` dedup set on `vendor_id` is a client-side workaround for potential duplicate rows in `order_schedule`. A comment explaining why duplicates can occur (e.g., a store has two schedule rows for the same vendor-day pairing) would help future readers understand why the dedup is needed rather than using `.distinct()` or a `UNIQUE` constraint.

- `src/hooks/useEodSubmit.ts:226-231` — The intentional `eslint-disable-next-line react-hooks/exhaustive-deps` for the mount-only drain effect is correct, but the comment says "Intentionally omitting `drain` and `isOnline` from deps." `drain` is stable (it's a `useCallback`), so omitting it would not actually cause stale closure issues. The comment could be more precise: the omission of `isOnline` is the meaningful one (to avoid re-running on every connectivity flip, which the other effect handles). Minor documentation issue.

- `src/screens/SignIn.tsx:134-137` — `accessibilityRole="header"` on the large title `Text` is correct, but the subtitle text immediately below (`accessibilityRole` is absent) says "Sign in" which is the screen's primary heading. The two texts have their roles swapped: the display text at line 135 reads `t('auth.signIn.subtitle')` which renders "I.M.R Staff" (the app name), and the smaller text at line 137 reads `t('auth.signIn.title')` which renders "Sign in". So `header` role on "I.M.R Staff" and no role on the call-to-action text is intentional — but the variable names `auth.signIn.title` and `auth.signIn.subtitle` are backwards relative to visual prominence (the `display`-size text is called `subtitle`). This is confusing to read even if the keys map to the right strings.

- `src/components/ListRow.tsx:29-36` — The `Wrap` variable is typed as `Pressable | View` via a conditional, but the style prop on `View` does not accept a function `({ pressed }) => [...]`, only `Pressable` does. This is papered over by the `({ pressed }: { pressed?: boolean } = {})` defaulted-to-empty-object cast at line 36. The code works at runtime because `View` ignores style functions it doesn't understand, but TypeScript should be flagging this. If it isn't, the `as unknown as` or structural duck-typing is masking the type error. Consider two explicit render paths instead of one polymorphic `Wrap`.
